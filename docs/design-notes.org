* Rationale: why build this                                        :rationale:
Motivation: not wanting to start a whole R session (plus =library(ggplot2)=,
easily 1–2 s) for a one-off plot. The claim "current command-line plotters
are not efficient and don't produce high-enough quality figures" was
verified [2026-07-18] with one nuance:

** The precise gap
No existing tool is both *fast-starting* and *ggplot-quality*. The
intersection is empty; tools split into two camps.

** Fast but ugly (the C camp)
- *gnuplot* :: actually very efficient (C, ms startup, millions of points,
  active — 6.0.3, June 2025). =pdfcairo= gives real vector PDF with Pango
  text, but defaults are dated, no layout engine (multiplot cropping bugs,
  =fontscale= silently 0.5 in pdfcairo), legends/facets/themes decades
  behind ggplot. Efficiency claim slightly unfair to gnuplot; quality
  claim holds.
- *GNU plotutils / libplot* :: effectively dead — last release 2.6,
  September 2009.
- *PLplot* :: alive but sleepy; Cairo/Pango drivers render text well, but
  1990s aesthetic, no layout/legend engine.
- *GR framework* :: fast C graphics (used as a matplotlib speed backend),
  but a graphics kernel, not a figure-composition system.
- *Terminal plotters* (youplot, plotext, =gnuplot dumb=) :: ASCII/sixel,
  not publication PDF.

** Beautiful but slow (the interpreter camp)
- *ggplot2* :: requires R startup + package load.
- *matplotlib / plotnine* :: same startup tax in Python. Closest prior
  art: p9-cli, a ~250-line CLI over plotnine, unmaintained.
- Notably: *no C implementation of the grammar of graphics exists*. The
  niche — =plot data.csv -x hp -y mpg --color cyl -o fig.pdf= returning a
  ggplot-grade PDF in ~20 ms — is unoccupied.

** Why ggplot output looks good (decomposed build cost)
1. /Text rendering & metrics/ — R's =cairo_pdf= draws via Cairo/Pango
   (shaping, kerning, subsetted embedded fonts). Free in C via
   =cairo_pdf_surface= (+ Pango later). Same stack R uses.
2. /A measuring layout engine/ — grid positions nothing by guesswork:
   labels, legends, strips are measured via font metrics and the panel
   rectangle is computed from what's left. Gnuplot's failures are mostly
   failures of exactly this.
3. /Good defaults as algorithms/ — pretty breaks are a published
   algorithm (extended Wilkinson; Talbot–Lin–Hanrahan 2010), plus scale
   expansion, hue/viridis palettes, theme defaults.
4. /The grammar itself/ — scales → geoms → guides → facets. Scope
   explodes here if allowed; a small grammar (point/line/bar/area/
   boxplot, color/size, one facet dim) covers most one-off plots.

** Feasibility estimate
MVP (CSV in, Cairo/Pango PDF out, scatter/line/bar, correct breaks,
measured legends, one clean theme) ≈ a few thousand lines of C; deps only
=cairo=, =pango=, =fontconfig=. Single-digit-ms startup, figure < 50 ms —
~100× latency win over Rscript with visually indistinguishable output.
Advice: take the Pango dependency; text shaping is the one part not worth
owning.

** Algorithm discoveries forced by the spike
1. Naive "pretty" breaks (round step to 1/2/5/10) are visibly wrong —
   produced only two y labels. Replaced with the real
   *Talbot–Lin–Hanrahan extended Wilkinson* algorithm (simplicity /
   coverage / density / legibility scoring, same as R's
   =labeling::extended()=) in ~50 lines of C; reproduces ggplot's
   choices exactly.
2. ggplot computes breaks on the *expanded* limits (data range + 5%),
   then drops breaks outside them — that's why hp gets 100/200/300
   (from candidate {0,100,200,300}) rather than 50-steps. Non-obvious,
   visually significant.

** Known gaps (deliberately deferred)
- Cairo "toy" text API instead of Pango — fine for numerals/Latin;
  Pango needed for markup, superscripts, non-Latin shaping.
- Legend key spacing eyeballed within ~1 pt.
- Point-size conversion (size 1.5 → 2.15 pt radius) empirical, not
  derived from grid's =fontsize = size × .pt + stroke × .stroke/2=.
- Design note: ggplot's default hue palette replicated for fidelity,
  but its red/green pair is not colorblind-safe — the real tool should
  make Okabe-Ito or viridis an easy default.

* Design: interface & layout engine                                 :design:
Decided [2026-07-18] after the layout-algorithm discussion.

** Interface: mini-DSL + CLI shortcuts
- One code path: CLI flags are pure sugar, desugared into DSL fragments
  (=flags → DSL string → parsed spec=). =--dump-spec= prints the
  desugared DSL for learnability.
- *DECIDED: verbatim ggplot2 vocabulary* (not a terse dialect):
  #+begin_example
  cplot 'data.csv
    + aes(hp, mpg, colour=factor(cyl))
    + geom_point()
    + geom_smooth(method="loess")
    + scale_y_log10()
    + facet_wrap(~gear)' -o fig.pdf
  #+end_example
  Rationale: R users can near-copy-paste; ggplot docs double as our docs.
- Parser implications of verbatim names: positional aes args (x, y),
  =factor()= calls inside aes, formula syntax =~gear=, string args,
  colour/color aliases. Still recursive descent (~400–600 lines);
  unsupported ggplot functions error with the supported-subset list.
- Flags for the 90% case: =-x -y -c -s -m -f -t --log x|y -o=; stdin is
  the default data source so it pipes.

** Layout engine: gtable, not a constraint solver
- ggplot2 uses no general solver — the figure is a table (gtable) with
  two unit kinds per row/col:
  - /fixed (pt)/: theme absolutes (margins, tick length) or *measured
    from content* via font metrics (widest y label, title height, legend
    block).
  - /null (flexible)/: weighted share of remaining space; only panels
    are null.
- Resolution = one arithmetic pass: sum fixed, subtract, split remainder
  among nulls. ~100 lines.
- Single-panel column layout:
  =[margin | y-title | y-labels | ticks | PANEL(null) | gap | legend | margin]=
- Faceting falls out free: more panel columns/rows; strips are measured
  rows; weight ≠ 1 gives =space="free"= later.
- Rendering: micro-grid — grob tagged union (rect/line/text/points/
  child-table) in viewports (affine transform + clip). Skip grid's
  nested rotated viewports and unit-arithmetic trees; rotation
  special-cased (y-title, angled x labels).
- Theme as data: text elements = {family, size, color, margins,
  hjust/vjust, angle}; measurer consumes theme structs, so themes are
  pure data.
- Legend-taller-than-figure policy deferred (ggplot clips; we could
  warn or multi-column).
- *DECIDED: fixed m=5 break target* (ggplot-style), keeping layout
  single-pass and deterministic; size-adaptive (matplotlib-style) breaks
  would need a ~2-pass fixed point — architecture permits, not planned.

** Pipeline
parse DSL → load CSV (typed cols) → facet split → train scales →
breaks/labels (extended Wilkinson on expanded limits) → MEASURE →
ASSEMBLE gtable → RESOLVE → RENDER.

** What the engine consists of
- =Unit= (fixed pt | null weight), =GTable= (rows/cols of Units +
  grob list), =gt_resolve()= — the whole resolver is ~30 lines: sum
  fixed, split remainder among nulls by weight.
- Grobs: tagged union rect | line | text | points | child-table,
  positioned in npc (0..1, y-up) within a cell *span*; per-grob clip
  flag (panel contents clip, tick labels don't).
- Text anchoring: hjust ∈ [0,1] × valign {top, bottom, ink-center} ×
  rot90; rotated y-title's /column width/ = the text's /font height/,
  which the table expresses naturally.
- Legend = a *nested GTable* placed as a grob (fixed-size child,
  centered in its cell) — composition works, which is the mechanism
  facet panels and patchwork-style plot grids will reuse.

** Notable properties confirmed
- Panel-aligned title falls out for free: the title grob just lives in
  the panel's column.
- The figure spec section of main() is now purely declarative: units +
  grob placements; no coordinate arithmetic outside the engine.
- Single resolve pass; deterministic; measured cells come from font
  metrics exactly as in spike 1.

* Exploration: heatmap support — port wheatmap semantics            :design:
Explored [2026-07-18] (~/repo/wheatmap, user's own CRAN package, 3.6k
lines R). Question: ggplot heatmaps vs ComplexHeatmap vs porting
wheatmap into cplot, adapted toward ggplot conventions.

** Why ggplot heatmaps underwhelm (user instinct confirmed)
geom_tile/geom_raster need matrix→long melting, no clustering/
dendrograms, no annotation-bar composition, panel+facet layout only.
ComplexHeatmap dominates bioinformatics for this reason but is huge
(10k+ lines, elaborate splits/annotations) and someone else's API.

** wheatmap model (from source reading)
- Canvas npc [0,1]^2; every object resolves to a WDim rect
  {left,bottom,width,height,nr,nc,row.split,column.split}.
- Placement closures TopOf/Beneath/RightOf/LeftOf(+corners,WPosition):
  default anchor = last object; size defaults PROPORTIONAL TO DATA
  (1/nr of the scale reference per row, clamped by min.ratio) — a
  1-row colorbar above a 40-row heatmap auto-gets 1/40 height.
  h.aln/v.aln (alignment ref) decoupled from h.scale/v.scale (sizing
  ref); pad in npc.
- Named objects, cross-reference by name (ggplot has no equivalent;
  patchwork is positional, not named).
- CMPar: brewer/colorspace/custom (jet,bwr) ramps, continuous via
  colorRamp interpolation, discrete label2color, dmin/dmax, na.color.
- Clustering: stats::hclust ward.D2 + euclidean; dendrogram drawing
  adapted from ComplexHeatmap.
- Auto-margin: print-time gtable wrap measures outward labels →
  reserves page margins (same idea as cplot's measurement pass).

** Key porting insight
wheatmap's layout is SIMPLER than our gtable: placements resolve to
plain npc rectangles. Render primitives = colored rect grids, text,
polyline dendrograms — all already in cplot's engine (G_RECT sub,
G_POLYLINE, G_TEXT, measurement). The two engines compose: gtable
outer (auto-margins), anchor-canvas inner as a child grob.

** Proposed ggplot-flavored surface (harmonization)
#+begin_example
cplot 'expr.csv
  + heatmap(name="main", cluster=both)          # WHeatmap + *.cluster
  + scale_fill_gradient2(low="blue", mid="white", high="red")
  + annotation(groups.csv, top_of("main"))      # WColorBarH
  + legend(right_of("main"))                    # WLegendV
  + dendrogram(left_of("main"))
  + labs(title="...")'
#+end_example
- CMPar → ggplot scale_fill_* vocabulary (viridis/brewer/gradient/
  gradient2; keep jet/bwr as legacy names)
- WCustomize → theme(); placements → snake_case top_of()/beneath()/
  right_of()/left_of() keeping pad/h.aln/v.scale semantics
- matrix mode: CSV with rownames in col 1; heatmap() in the spec
  switches the DSL into matrix mode

** New C pieces needed (effort)
- matrix CSV reader (small); anchor-placement resolver (trivial —
  the R closures are ~30-line C functions over rects)
- continuous colormaps: ramp interpolation + embedded viridis table,
  key brewer ramps, jet/bwr (moderate)
- continuous colorbar legend — reuses extended Wilkinson breaks
- hclust ward.D2 + euclidean + dendrogram layout (biggest piece,
  ~200-300 lines, O(n^2..3) naive fine for heatmap sizes)
- raster mode for big matrices: embedded image w/ NEAREST filter
  (geom_raster equivalent; keeps PDF small)

** Architecture decision: canvas and gtable stay separate layers
Q [2026-07-18]: does the anchor canvas make gtable messy / should
wheatmap be compiled INTO gtable rows-cols? Decision: keep two
assemblers over ONE grob/render layer; do not unify layouts.
- Anchor mosaics don't share global cut lines (README example is not
  a lattice); compiling to a table = union-of-cuts lattice with many
  spanning cells, all fixed fractions — gtable's measured/null unit
  semantics buys nothing there; WPosition can't be a table at all.
- Precedent: wheatmap R itself is free canvas inside + gtable wrap
  outside (auto_margin.R). Same split here: heatmap mode = trivial
  gtable [measured margins | canvas(null) | measured margins]; the
  canvas assembler resolves anchors to rects and emits ordinary grobs
  into that one cell. gtable engine unchanged (maybe +G_IMAGE later).
- Mutual nesting already mechanically supported: gt_resolve takes any
  rect (gtable-in-canvas, e.g. legends as child tables); a canvas is
  just grobs in a cell (canvas-in-gtable).
- Rule to keep it clean: gtable never learns about anchors; the
  canvas never learns about units. Legends/theme/text measurement
  shared. Wrinkle to document: wheatmap pads are npc fractions,
  chrome stays in pt.

* Milestone: heatmap spike 2 — clustering + dendrograms            :spike:
Result [2026-07-18], in the cinderplot repo. hclust + dendrograms
land; the anchor-canvas assembler gained clustering with no engine
changes (dendrogram segments are ordinary G_LINE grobs).

** New pieces
- =src/cluster.c= :: R-compatible hclust. Euclidean dist (R's NA
  pairwise scaling p/#complete), Ward.D2 via Lance-Williams on squared
  distances, R merge convention (singletons-before-clusters, ascending
  id), leaf order by recursive first-then-second expansion. Naive
  O(n^3), fine for heatmap sizes.
- =tests/cluster_check.c= :: dumps order+heights for R comparison.
- DSL: heatmap(cluster=rows|cols|both|none); dendrogram(placement)
  (left/right = row tree, top/beneath = column tree).

** Ergonomic win over wheatmap R
Anchored objects INHERIT the clustered order automatically: a
top/beneath annotation inherits the anchor's column order, a
left/right annotation its row order, and dendrogram() pulls the
anchor's tree + order. No manual reordering (wheatmap R makes you pass
grp[clust$order]). Implemented via per-heatmap display->data maps
(roword/coword), identity when unclustered — so unclustered output is
byte-unchanged (composite regression 0 px).

** Verified
- C order + merge heights == R hclust(dist(m),"ward.D2") EXACTLY for
  expr.csv rows (20) and cols (12).
- Clustered heatmap + both dendrograms + inherited annotation matches
  wheatmap R (both.cluster): same column block pattern 2/3/2/1/4 (hue
  differs: our ggplot palette vs Set1). Dendrogram topology matches;
  our right-angle brackets render cleaner than wheatmap's.
- Regression: unclustered composite 0 px diff; grammar-mode test suite
  passes; dendrogram-without-cluster errors clearly.

** Deferred (spike 3 and beyond)
- Auto-margin measurement (fixed 30pt legend slack, no row/col tick

- Dendrogram depth default 0.2*anchor (wheatmap uses its own); width=/

* Milestone: heatmap spike 3 — auto-margins + row/col labels        :spike:
Result [2026-07-18]. wheatmap's auto_margin ported: outward labels are
measured and reserved as page margins so nothing clips.

** New pieces
- DSL: heatmap(rownames=left|right|none, colnames=top|bottom|none).
  Default none (opt-in, like wheatmap yticklabels=FALSE).
- heatmap.c: a measurement pass computes marL/marR/marT/marB in pt from
  every OUTWARD-facing label — rownames (L/R, horizontal), colnames
  (T/B, rotated 90 via existing rot90 grob), legend colorbar tick
  labels — replacing the old fixed 30pt legend slack. Boundary test: a
  label reserves margin only if its object's edge on that side is at
  the normalized bbox boundary (0 or 1), so interior labels don't
  inflate the page.
- Labels drawn into the reserved margin by placing grobs at npc just
  outside [0,1], converting the pt gap through the FINAL canvas cell
  size (cw_pt/ch_pt). Labels follow the clustered display order
  (rn[roword[slot]]).

** Coupling avoided
Boundary-anchored labels have pt extent independent of canvas size, so
margins resolve in a single pass — no measure/resolve iteration. (Grid
gets this from draw-time measurement; we get it from the bbox-boundary
restriction.)

** Verified
- Long-rowname clustered heatmap (very_long_gene_name_NN right +
  sample_NN bottom + both dendrograms): all labels fully visible, none
  clipped; right/bottom margins auto-sized to the longest label.
- Regression: no-legend/no-label heatmap byte-identical (0 px) — all
  margins collapse to MARGIN, same geometry as before. Legend figures
  change intentionally (measured margin < old fixed 30pt). Grammar mode
  and test suite unaffected.

** Deferred
- rownames + legend on the SAME side (would overlap): unsupported;
  wheatmap shifts siblings outward — not ported. Put them on
  different sides.
- colnames=top combined with a title can collide (both want the top).
- Raster mode (>200k cells); horizontal legends; per-object colormaps;
  45-degree colname rotation.

* Milestone: rigid legends (V + H), decoupled from the matrix       :spike:
Result [2026-07-18]. Fixed a design flaw: the spike-1 legend coupled
its size to the heatmap (height = anchor height, width = 0.05 npc) —
too tall on big matrices, too short on small ones, and it drifted with
figure size. Redesigned per ComplexHeatmap: legends are RIGID chrome at
a fixed physical size, never coupled to matrix dimensions.

** Design
- Fixed pt dims (ComplexHeatmap defaults): bar 4mm thick, 28mm long.
  =#define LEG_BAR (4*MM)=, =LEG_LEN (28*MM)=, MM = 72/25.4.
- Legend pulled OUT of the canvas bbox (no longer a normalized canvas
  rect). It's margin chrome, like axis labels: reserves a fixed-size
  margin on its placement side and is drawn there, centered on the
  target heatmap's cross-axis.
- Both orientations now: vertical (right_of/left_of) reserves L/R
  margin; horizontal (beneath/top_of) reserves B/T margin. Ticks and
  labels flip to the outward side automatically. (Answers "LegendV and
  LegendH?" — both now exist; before, horizontal errored.)

** Verified
- Rigidity: colorbar renders 165 px (=28mm at 150dpi) for BOTH a 2-row
  and a 20-row matrix at the same figure size — identical. That's the
  bug fixed: size no longer tracks the matrix.
- No-legend heatmap still byte-identical (0 px). Test suite passes.
  Labels-right + legend-beneath compose cleanly (different sides).

** Deferred
- Discrete/categorical legends (fixed 4mm grid squares + gap) not yet —
  only continuous colorbars. Annotations still self-color without a key.
- Legend title (labs(fill=)) not drawn yet.
- Legend + same-side rownames would overlap (put on different sides).
- Width/height are fixed; no pt override exposed in the DSL yet.

* Milestone: discrete legends + legend titles + stacking            :spike:
Result [2026-07-18]. Categorical legends and titles added; multiple
legends now pack in the margin instead of overlapping.

** New
- Discrete (categorical) legend: legend() anchored to a categorical
  annotation draws a key of fixed 4mm colour squares + level labels,
  reusing the annotation's factor + hue palette (colours match the
  annotation bar exactly). Rigid, like the colorbar. Vertical only
  (horizontal discrete errors clearly).
- Legend titles: legend(title="...") explicit; else default from
  labs(fill=) (continuous) or the annotation column name (discrete);
  none if unset. Drawn at SZ_BASE above the legend, left-aligned.
- Legend STACKING: vertical legends on a side are packed top-to-bottom,
  centred as a group on the canvas (ComplexHeatmap-style), so a
  colorbar + one or more annotation keys coexist without overlap. Fixes
  the earlier bug where a discrete legend centred on a thin annotation
  strip clipped off-canvas.

** Design touch
- draw_one_legend() factored out; a per-side stacking driver computes
  each block's height (content + title) and lays them out. Discrete
  key colours pulled from the stored ann_f/ann_pal on the annotation
  RObj.

** Verified
- expr colorbar + group (A/B/C) key stack cleanly in the right margin;
  discrete colours match the top annotation bar; all levels visible;
  both titled.
- No-legend heatmap still byte-identical (0 px). Test suite passes.
  Single vertical legend unchanged; discrete+horizontal errors.

** Deferred
- Legend title (labs(fill=)) needs the annotation named for a discrete
  legend to anchor (auto-names are h2, h3...). Consider anchoring by
  data/column later.
- Numeric (continuous) annotations get no legend yet (own scale).
- Horizontal discrete legends; per-object fill scales; legend title
  centring options.

* Milestone: anchor-by-column + continuous annotation legends       :spike:
Result [2026-07-18]. Two ergonomics from the legend review.

** Anchor a legend (or any object) by an annotation's data column name
- find_obj() now resolves an anchor string as: exact object name= first,
  then an annotation's source column name. So legend(right_of("group"))
  works without naming the annotation; the auto-name h2 still works too.
- Protective: column names match as EXACT bytes, and a name with spaces
  or punctuation is fine when quoted in the DSL — right_of("my group
  (v2)") resolves (raw_token -> string_lit handles the quotes). Explicit
  name= wins over a colliding column name; first match wins on dup
  columns; unknown anchors still error clearly.

** Continuous (numeric) annotation legends
- A numeric annotation now stores its OWN scale (viridis over its
  min/max) on the RObj (ann_continuous/ann_dmin/ann_dmax/ann_fill).
- legend() anchored to a numeric annotation draws a colorbar in that
  annotation's own scale, not the heatmap's shared fill scale
  (legend_scale() picks the source's range + colormap; both the margin
  measurement and the draw use it). Colours match the annotation bar.
- Default title for any annotation legend = its column name.

** Verified
- One figure: expr heatmap colorbar + group discrete key + score
  numeric colorbar (own 0-75 viridis scale), all three stacked in the
  right margin, all anchored by column name (no name=).
- Quoted weird column name resolves; unknown anchor errors; no-legend
  output byte-identical (0 px); test suite passes.

** Deferred
- Ambiguous duplicate column names: first wins (could warn).
- Horizontal discrete legends; per-heatmap fill overrides.

* Milestone: raster mode for big heatmaps                           :spike:
Result [2026-07-18]. Large heatmap bodies render as a single embedded
image instead of N vector rectangles, so PDF size and draw time stay
flat. Replaces the old hard error at >200k cells.

** New
- G_IMAGE grob (gtable.c): paints a cairo ARGB32 image surface into a
  cell sub-rect, scaled with CAIRO_FILTER_NEAREST so cell edges stay
  crisp (the geom_raster trick). Engine gains one grob type; layout
  unchanged.
- heatmap.c: a heatmap body with > RASTER_MIN_CELLS (2000) cells is
  filled into an ARGB buffer (one pixel per display cell, in clustered
  order) and emitted as one G_IMAGE; below that it stays vector, so
  small heatmaps are byte-identical. col_argb() quantizes Col ->
  0xAARRGGBB. Sanity ceiling RASTER_MAX_CELLS = 25M (~100 MB buffer).

** Verified
- Small heatmap (240 cells) byte-identical to golden (0 px); test
  suite passes.
- 120,000-cell clustered heatmap + legend: 0.09 s, ~200 KB PDF (would
  have errored before). Row-gradient check confirms orientation (row 0
  at top) and value->colour mapping are correct under raster.

** Deferred
- Annotations still vector: a huge row/col annotation next to a big
  heatmap is many rects. Same G_IMAGE technique applies; do when
  needed.
- Raster uses one pixel per cell (nearest-scaled); no anti-aliased
  downsampling for matrices far larger than the panel in pixels.
